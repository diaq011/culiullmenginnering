## 一、本节课所需的基础理论

### 1. 本节课中的 Prompt 工程含义

本节课中，Prompt 工程不被理解为“写几句给模型的话”，  
而是被定义为：

**把任务、上下文、输出格式和约束组织清楚，使模型能够较稳定地完成具体任务。**

### 2. 本节课最关键的 4 个要点

1. 任务描述清晰
2. 上下文信息充分
3. 输出格式固定
4. 结果需要持续验证和优化

提示词工程参考资料：https://www.promptingguide.ai/zh

---

## 二、本节课需要开发的内容

### 项目名称

`game_suggester`

### 功能目标

输入一个房间号，读取当前游戏状态，并给出“下一步建议”。

### 最低建议输出

- 推荐落子位置
- 推荐输入值
- 推荐理由
- 风险提示
- 置信度

---

## 三、本节课必须使用的两个接口

接口来源：

- `game_coach_demo/game_coach_game`

开发中必须使用：

1. `GET /api/coach/snapshot/<room_code>`
   - 用于获取当前对局快照
2. `POST /api/coach/evaluate_move`
   - 用于评估某一步是否合法及其可能影响

本节课的 `suggester` 需要基于这两个接口生成建议，而不是脱离当前游戏状态单独输出结果。

## 五、本节课建议开发步骤

## 步骤 1：先运行游戏

进入：

```bash
cd game_coach_demo/game_coach_game
python app.py
```

用浏览器打开游戏页面，完成登录并进入房间。

## 步骤 2：编写 `suggester` 页面或接口

可以采用以下任一形式：

- 一个简单网页
- 一个简单 Flask 接口页面

最低要求为：

- 可以输入房间号
- 可以显示建议结果

## 步骤 3：获取快照

至少获取以下信息：

- 当前棋盘
- 当前回合
- 当前分数

## 步骤 4：先生成候选建议

先让模型基于快照和规则给出 1 个或多个候选建议。

## 步骤 5：再评估候选走法

对模型给出的候选建议，  
再使用 `evaluate_move` 判断是否合法以及可能产生的影响。

## 步骤 6：必要时修正建议

如果评估结果不理想，可修改 Prompt 或重新生成候选建议。

## 步骤 7：展示最终建议

将最终版本展示在页面上，并说明理由与风险。

## Prompt 最低应包含的要素

1. 角色：游戏建议助手
2. 任务：给出下一步建议
3. 游戏状态：来自 `snapshot`
4. 输出格式：固定字段
5. 展示前需经过 `evaluate_move` 验证

## 模型要求

模型建议使用：

- `qwen3.5:4b`

也可使用其他本地可用模型。

## 测试要求

至少完成两轮测试：

1. 自测
2. 与其他同学或小组交叉测试
